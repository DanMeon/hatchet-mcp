# v0.3.0 ‚Äî Operational Toolkit Expansion

This version closes the largest gap between what `hatchet-sdk` exposes and what `hatchet-mcp` surfaces, then tightens the supporting surfaces (HTTP transport, error structure, stderr telemetry, MCP annotations). The latest GA (`v0.2.1`) carries 25 read + 17 mutating tools and a structured stderr emitter from v0.2.0. Building on the unreleased `minimal_output` default flip for `list_runs` / `list_events` already on `main` (#5), this spec adds 10 read + 4 mutating tools (one new domain `webhooks`, one new domain `meta`), reuses the cached `ApiClient` across REST-direct calls, typed API errors with `status` / `kind`, plumbs the JSON-RPC `request_id` into every `tool.ok` / `tool.error` record, fixes a `v0.2.1` production bug where `resources.py` bypassed the reliability wrapper, and corrects one invariant violation in `get_run_status` (snake_case ‚Üí camelCase return key ‚Äî the only breaking change in the release).

Existing invariants are preserved: read-only by default, the two-layer mutation gate, the 500-item bulk caps (run and event replay; new scheduled-delete cap), the `redact()` boundary, and the camelCase REST output shape. The mutation gate now also guards 4 new mutating tools; the destructive-annotation invariant on every mutating tool is unchanged.

Rationale, alternatives, and failure scenarios for the key decisions live in the paired ADR: [operational-toolkit-expansion-research.md](../../design/v0.3.0/operational-toolkit-expansion-research.md).

## Decisions

| Item | Value | Rationale |
|---|---|---|
| 1 ‚Äî Tool surface expansion | 10 new read tools (`get_workflow_version`, `get_workflow_metrics`, `get_task_stats`, `get_scheduled`, `list_dag_tasks`, `list_webhooks`, `get_webhook`, `top_failing_workflows`, `list_stuck_runs`, `describe_run_failure`) + 4 new mutating tools (`cancel_run`, `replay_run`, `replay_events`, `bulk_delete_scheduled`); one new domain module `tools/webhooks.py` (V1 inbound webhooks, read-only ‚Äî credentials in create/update kept out of MCP), one new domain module `tools/meta.py` (aggregation/composition over existing reads) | Single-PR coverage of every SDK feature-client method that maps cleanly to a read/mutating contract; `bulk_delete_scheduled` mirrors the v0.2.0 `cancel_runs`/`replay_runs` safety posture (dry-run default, mutex IDs/filter, 500-cap on explicit IDs); `replay_events` enforces a 500-event cap client-side before the REST call. See ADR ¬ß1 for the full surface inventory + the deliberate exclusions (`put_rate_limit` ‚Äî gRPC under the hood; `delete_workflow` ‚Äî irreversible, future elicitation gate). |
| 2 ‚Äî Meta-tool pattern | `top_failing_workflows(since, limit, scan_limit)` ranks workflows by FAILED count over a window; `list_stuck_runs(stuck_after_minutes, scan_limit)` filters RUNNING runs by elapsed minutes; `describe_run_failure(run_id, log_tail)` issues `get_run` ‚Üí `(logs / events / timings)` via `asyncio.gather(return_exceptions=True)` and returns one assembled payload. No new Hatchet endpoint; pure aggregation over `aio_list` / `aio_get_task_run` / `aio_list (logs)` / `v1_task_event_list` / `v1_workflow_run_get_timings`. The timings branch is best-effort ‚Äî if it raises `ApiException`, `HatchetAPIError`, or `asyncio.TimeoutError` the result demotes to `timings=null` instead of failing the whole call | The three queries are the dominant operator/oncall round-trips against this MCP today; collapsing each into one tool call removes a 3‚Äì5-step LLM chain. The `truncated` flag exposes scan-limit honesty so a 1000-row failure burst does not silently undercount. See ADR ¬ß2. |
| 3 ‚Äî HTTP connection reuse | One process-wide cached `ApiClient` via `client.get_api_client()`, guarded by `threading.Lock` for the double-checked init (because `_rest_call` dispatches via `asyncio.to_thread`); `_rest_call` no longer enters the SDK's `with base.client() as client:` context ‚Äî `ApiClient.__exit__` is a no-op so skipping it leaks nothing, the underlying `urllib3.PoolManager` is documented thread-safe, and keep-alive now amortizes TCP+TLS across the 11 REST-direct sites (events, observability, tasks, runs filter-resolve, etc.) | Each `BaseRestClient.client()` call constructed a new `ApiClient` ‚Üí `RESTClientObject` ‚Üí `PoolManager` per request, paying a TCP+TLS handshake per call. With cached pool reuse this drops to amortized zero. The race in the init (two `asyncio.to_thread` workers can both see `_api_client is None`) is closed by a `threading.Lock` rather than left for the GC to merge. See ADR ¬ß3. |
| 4 ‚Äî Resources reliability wrap (production bug) | All 6 resource handlers in `resources.py` now go through `_reliability_wrap(fn, retry=True)` (created once at module import time); `resources.register()` uses the wrapped versions. Before this fix, resource reads bypassed the 30s deadline, retry on 5xx/429/transport errors, and the `tool.ok` / `tool.error` log records ‚Äî a hung Hatchet response could hang the MCP session indefinitely with no audit trail | The reliability wrapper was added at v0.2.0 and applied at tool registration only (`server.py:register_read_tools`); `resources.py` imported the raw handler functions and called them directly. A real production bug present since `v0.2.0`, surfaced during this release's review pass. See ADR ¬ß4. |
| 5 ‚Äî MCP annotation hygiene | Every read tool registration carries `ToolAnnotations(readOnlyHint=True, destructiveHint=False, idempotentHint=True, openWorldHint=True)` returned by the `_read_only_annotations()` **factory** (not a shared singleton ‚Äî Pydantic V2 `ToolAnnotations` is mutable and a stray write to one would flip the hint on all 35 read tools); every tool (read + mutating) now also advertises the MCP 2025-06-18 `title` field via `_humanize_tool_name(name)` (snake_case ‚Üí Title Case) | Clients that honor annotations (Cursor, Claude Code in trusted-server mode) skip per-call approval prompts when `readOnlyHint=True` ‚Äî without the explicit hint, the spec default is "destructive, not read-only" and clients prompt on every read. `title` populates the human-readable name in MCP UIs that show it separately from `name`. See ADR ¬ß5. |
| 6 ‚Äî Error + observability structure | `_api_error()` now returns `HatchetAPIError(message, status: int \| None, kind: Literal["validation_error", "unauthorized", "not_found", "conflict", "rate_limited", "server_error", "unknown"])`, a `RuntimeError` subclass so existing `pytest.raises(RuntimeError, match="Hatchet API error")` callers still match. `_reliability_wrap` extracts the MCP JSON-RPC `request_id` via `mcp.get_context().request_id` (lazy import, returns `None` outside a request) and adds it to every `tool.ok` / `tool.error` record; `tool.error` also carries `error_status` (int) and `error_kind` (str) from the `HatchetAPIError` attributes | The v0.2.x stderr records were per-tool but anonymous ‚Äî under concurrent stdio tool calls there was no way to attribute a record to a JSON-RPC request. The error message also forced LLM consumers to string-match `"status 429"` to pick a recovery path. Both are mechanical fixes that unblock structured downstream consumers. See ADR ¬ß6. |
| 7 ‚Äî Breaking ‚Äî `get_run_status` camelCase return key | Return key flips from `{"workflow_run_id": "..."}` to `{"workflowRunId": "..."}`; the resource at `hatchet://runs/{workflow_run_id}/status` follows by pass-through. The path parameter name (`workflow_run_id`) is unchanged | CLAUDE.md invariant 4 mandates Hatchet REST shape (`by_alias=True` camelCase) for every tool output; the hand-built snake_case dict in `runs.py:get_run_status` was an inconsistency introduced before the invariant was enforced. Fixing it is breaking but the surface is tiny (one key in one tool) and the inconsistency was actively misleading LLMs that learned the camelCase shape from `get_run`, `list_runs`, etc. See ADR ¬ß7. |

## Acceptance Criteria

- **AC-1** ‚Äî `len(server.READ_TOOLS) == 35` and `len(server.MUTATING_TOOLS) == 21`; every catalog entry's handler is callable and the module is registered in `_TOOL_MODULES`
- **AC-2** ‚Äî `top_failing_workflows` returns rows sorted by `failCount` descending, sets `truncated=true` when `scanned >= scan_limit`; `list_stuck_runs` returns only runs whose `(now ‚àí started_at)` exceeds `stuck_after_minutes`; `describe_run_failure` returns a single payload containing `failingTask`, `logs`, `events`, `timings` (timings may be `null` when the timings REST call raises `ApiException` / `HatchetAPIError` / `asyncio.TimeoutError`)
- **AC-3** ‚Äî calling `get_api_client()` ten times concurrently via `asyncio.gather(*(asyncio.to_thread(get_api_client) for _ in range(10)))` produces one `ApiClient` instance (verified by `id()` identity and constructor call count)
- **AC-4** ‚Äî a Hatchet 503 on a resource read (e.g. `hatchet://workflows/{id}`) triggers the reliability wrapper's retry loop and emits a `tool.ok` / `tool.error` record on stderr (the wrap path is now identical to the corresponding tool call)
- **AC-5** ‚Äî every read tool registered on `app.mcp` advertises `annotations.readOnlyHint == True`, `annotations.destructiveHint == False`, and `tool.title` is a non-empty string with no `_` characters; the `_read_only_annotations()` factory returns a fresh `ToolAnnotations` instance per call (not a shared singleton)
- **AC-6** ‚Äî an SDK `ApiException(status=429)` raised inside any tool surfaces as a `HatchetAPIError` with `status == 429` and `kind == "rate_limited"`; the resulting `tool.error` stderr record contains `request_id` (present, may be `None` outside a request scope), `error_status == 429`, and `error_kind == "rate_limited"`. 4xx ‚Üí `kind ‚àà {"validation_error", "unauthorized", "not_found", "conflict", "rate_limited"}`; 5xx ‚Üí `kind == "server_error"`
- **AC-7** ‚Äî calling `get_run_status(workflow_run_id="run-xyz")` returns `{"workflowRunId": "run-xyz", "status": "<v1 status value>"}` (the key is camelCase `workflowRunId`, not snake_case `workflow_run_id`); the resource `hatchet://runs/run-xyz/status` returns a payload byte-identical to that tool's response

## Non-Goals

- **`outputSchema` populated from SDK Pydantic models for every tool** ‚Äî FastMCP auto-derives from the function return annotation, but our handlers return `dict[str, Any]` for `_dump` / minimal-output / aggregation reasons; populating proper schemas means TypedDict per handler or `add_tool(output_schema=...)` per registration site, both larger than this spec's surface. Deferred to a follow-up minor
- **MCP tool-argument completions** ‚Äî the MCP 2025-11-25 spec's `CompleteRequest.ref` accepts only `PromptReference | ResourceTemplateReference`; tool argument auto-completion is not a protocol capability today, so adding `mcp.completion(...)` here would only affect prompts/resource templates and would not reduce the LLM "wrong UUID" failure mode
- **`put_rate_limit` mutating tool** ‚Äî `Hatchet.rate_limits.aio_put` is implemented over gRPC (not REST); same reason `push_event` uses the legacy REST `event:create` instead of the SDK's gRPC event push (see `tools/events.py:147` comment). REST-only is an explicit project invariant; gRPC entry points wait until that invariant changes
- **`delete_workflow`** ‚Äî SDK marks it "DANGEROUS" (permanently deletes the workflow definition + all run history). Wiring it without an explicit confirmation gate would be a foot-cannon; deferred until MCP elicitation is supported by major clients (Claude Code feature request still open)
- **Tenant admin / API token / SNS / Slack / GitHub integration / member-invite tools** ‚Äî these are dashboard concerns, not orchestration ops; exposing them risks credential leaks and adds surface a typical MCP user never needs
- **Multi-tenant single-server deployment** ‚Äî explicit `init/multitenancy-and-dependencies.md` topic; orthogonal to the toolkit-expansion narrative
